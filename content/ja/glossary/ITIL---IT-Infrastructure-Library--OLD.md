## 定義と価値提案

**ITIL(Information Technology Infrastructure Library)**は、ITサービスマネジメント(ITSM)のための世界的に採用されているフレームワークであり、ITサービスの体系的な設計、提供、運用、継続的改善のためのベストプラクティスを提供します。ITILは標準化されたガイダンスを提供し、組織が以下を実現できるようにします。

- ITサービスをビジネス戦略および目標と整合させる
- 効率性を高め、運用コストを最小化する
- サービス品質と顧客満足度を向上させる
- リスクを管理し、規制コンプライアンスを確保する
- デジタルトランスフォーメーションと俊敏性をサポートする

ITILはベンダー中立であり、あらゆるセクターに適応可能で、国際的に最も広く使用されているITSMフレームワークです。([ITIL Official](https://www.itil.com/)、[PeopleCert ITIL](https://www.peoplecert.org/Frameworks-Professionals/ITIL-framework)、[Wikipedia ITIL](https://en.wikipedia.org/wiki/ITIL))

## ITILとは何か?

ITILは、ITサービスをそのライフサイクル全体を通じて管理するための構造化されたアプローチと推奨事項のセットです。以下のための包括的なベストプラクティスを提供します。

- **ITとビジネスニーズの整合:**ITがビジネスに価値を提供し、ビジネス目標をサポートすることを保証します。
- **サービス提供の標準化:**ITサービスの計画、提供、保守のための一貫したプロセスを確立します。
- **効率性の向上:**無駄を削減し、リソースの使用を最適化し、運用を合理化します。
- **継続的改善のサポート:**組織がITサービスを定期的に評価し、強化することを奨励します。

ITILはあらゆる規模と業界の組織で適用され、他のフレームワーク(例:Agile、DevOps、COBIT、ISO 20000)と頻繁に統合されます。([ITIL Process Wiki](https://wiki.en.it-processmaps.com/index.php/ITIL_4)、[Atlassian ITIL Guide](https://www.atlassian.com/itsm/itil)、[Ivanti ITIL Guide](https://www.ivanti.com/glossary/itil))

## ITILの歴史と進化

### 主要なマイルストーン

| 年      | バージョン    | 注目すべき特徴と変更点                                                                                         |
|-----------|------------|----------------------------------------------------------------------------------------------------------------------|
| 1989      | ITIL v1    | 英国中央コンピュータ・通信局(CCTA)により開発。ITプラクティスをカバーする40巻以上。    |
| 2001      | ITIL v2    | 9つの主要な出版物に合理化され、使いやすさのためにサービスサポートとサービスデリバリーに焦点を当てる。          |
| 2007      | ITIL v3    | ITサービスライフサイクルを導入:サービス戦略、設計、移行、運用、継続的改善。      |
| 2011      | ITIL 2011  | v3の更新により一貫性、エラー修正、明確性を向上。ライフサイクルと26のプロセス/機能。           |
| 2019      | ITIL 4     | サービスバリューシステム(SVS)を導入し、34の管理プラクティスに拡大。Agile、DevOps、Leanとの統合。     |

**所有権:**  
当初は英国政府機関により開発され、所有権と管理はAXELOS(英国内閣府とCapita PLCの合弁会社)に移行し、現在は認定と継続的な更新のためにPeopleCertが管理しています。([ITIL Wiki](https://en.wikipedia.org/wiki/ITIL)、[Process Wiki History](https://wiki.en.it-processmaps.com/index.php/History_of_ITIL))

## ITILフレームワークの中核コンポーネント

### ITIL v3サービスライフサイクル(現在も広く使用されている)

1. **サービス戦略:**サービスマネジメントのフレームワークとポリシーを確立し、市場の定義、オファリングの開発、財務管理を含む。
2. **サービス設計:**本番環境への導入のために新規または変更されたサービスを計画・設計し、要件、アーキテクチャ、プロセスに焦点を当てる。
3. **サービス移行:**設計から運用への移行を管理し、変更管理、リリース、展開を含む。
4. **サービス運用:**インシデント管理、問題管理、運用制御を含む、ITサービスの日常的な管理とサポートを処理する。
5. **継続的サービス改善(CSI):**サービスとプロセスの測定、レビュー、改善のサイクルを実装する。

各段階には、組織が高品質のITサービスを提供するためのガイドとなる特定のプロセスと機能が含まれています。([ITIL v3 Lifecycle](https://wiki.en.it-processmaps.com/index.php/ITIL_Processes)、[Atlassian ITIL Guide](https://www.atlassian.com/itsm/itil))

## ITIL 4:サービスバリューシステムと指針となる原則

### サービスバリューシステム(SVS)

SVSはITIL 4の中核であり、組織のすべての部分がIT対応サービスを通じて価値を創造するためにどのように連携するかを定義します。SVSコンポーネント:

- **機会/需要:**価値創造活動のトリガー。
- **指針となる原則:**効果的な行動と意思決定のための中核的な推奨事項。
- **ガバナンス:**組織の監督、ポリシー、方向性。
- **サービスバリューチェーン:**サービスの作成、提供、継続的改善を導く運用モデル。
- **プラクティス:**以前の「プロセス」に代わり、作業を実行するための組織リソースのセットを提供する。
- **継続的改善:**改善機会を特定し、行動する継続的なプロセス。

([IT Process Wiki on ITIL 4](https://wiki.en.it-processmaps.com/index.php/ITIL_4)、[ITSM.tools ITIL 4 Explained](https://itsm.tools/itil-4-explained/))

### サービスマネジメントの4つの側面

1. **組織と人材:**文化、役割、スキル、能力。
2. **情報と技術:**サービスをサポートするデータ、情報、技術、ツール。
3. **パートナーとサプライヤー:**第三者との関係と合意。
4. **バリューストリームとプロセス:**価値を提供する活動とワークフロー。

### 7つの指針となる原則

| 原則                          | 説明                                                                                       |
|-------------------------------------|---------------------------------------------------------------------------------------------------|
| 価値に焦点を当てる                     | すべてがステークホルダーに価値を提供すべきである。                                                   |
| 現状から始める                | 変更を行う前に現在の状態を評価する。                                                    |
| フィードバックを伴う反復的な進歩 | 小さなステップで変更を行い、各段階でフィードバックを使用する。                                         |
| 協力と可視性の促進 | チームとステークホルダー間の協力と透明性を促進する。                               |
| 全体的に考え、働く        | エンドツーエンドの価値提供のために、サービスを相互接続されたシステムとして扱う。                     |
| シンプルで実用的に保つ       | 不必要な複雑さを避け、必要なものだけを提供する。                                         |
| 最適化と自動化              | 効率を最大化し、手作業を削減するために、可能な限りプロセスを合理化し、自動化する。  |

各原則は普遍的に適用可能であるように設計されており、継続的改善とビジネス目標との強力な整合をサポートします。([IT Process Wiki Guiding Principles](https://wiki.en.it-processmaps.com/index.php/ITIL_4#Service_value_system))

## ITILのプラクティスとプロセス

ITIL 4は**34の管理プラクティス**を定義し、以下のように分類されます。

### 1. 一般管理プラクティス(14)

- 戦略管理
- ポートフォリオ管理
- アーキテクチャ管理
- サービス財務管理
- 労働力と人材管理
- 継続的改善
- 測定とレポート
- リスク管理
- 情報セキュリティ管理
- ナレッジ管理
- 組織変更管理
- プロジェクト管理
- 関係管理
- サプライヤー管理

### 2. サービス管理プラクティス(17)

- ビジネス分析
- サービスカタログ管理
- サービス設計
- サービスレベル管理
- 可用性管理
- キャパシティとパフォーマンス管理
- サービス継続性管理
- 監視とイベント管理
- サービスデスク
- インシデント管理
- サービスリクエスト管理
- 問題管理
- リリース管理
- 変更実現
- サービス検証とテスト
- サービス構成管理
- IT資産管理

### 3. 技術管理プラクティス(3)

- 展開管理
- インフラストラクチャとプラットフォーム管理
- ソフトウェア開発と管理

**主要なプラクティスの例:**

- **インシデント管理:**計画外の中断後に通常の運用を復元し、ビジネスへの影響を最小化する。
- **問題管理:**繰り返し発生するインシデントの根本原因を特定し、除去し、将来の中断を削減する。
- **構成管理データベース(CMDB):**IT資産とその相互関係を追跡・管理するための中央リポジトリ。

ITIL 4のプラクティスモデルは以前のバージョンよりも規範的ではなく、柔軟性と他のフレームワークや方法論との統合を重視しています。([IT Process Wiki on Practices](https://wiki.en.it-processmaps.com/index.php/ITIL_4#ITIL_4_management_practices))

## ITIL対ITSM

| 側面                | ITIL                                             | ITSM                                                            |
|-----------------------|--------------------------------------------------|-----------------------------------------------------------------|
| 定義            | ITサービスマネジメントのためのベストプラクティスフレームワーク| ITサービスを管理するための規律または方法論              |
| 範囲                 | 詳細なガイダンス、プロセス、原則     | ITサービスの設計、提供、管理、改善のためのすべての活動 |
| 関係          | ITILはITSMを実装するためのフレームワーク            | ITSMはITIL、COBIT、ISO 20000などを使用して実装可能      |
| 例               | ITILインシデント管理プロセス                 | ITSMはインシデント、問題、変更、構成、資産管理などをカバー |

ITILはITSMを実装するための主要なフレームワークですが、ITSMはITをサービスとして管理するより広範な規律です。([Atlassian ITIL/ITSM Guide](https://www.atlassian.com/itsm/itil)、[TechTarget ITIL Guide](https://www.techtarget.com/searchdatacenter/definition/ITIL))

## ITIL認定構造

### ITIL 4認定スキーム

| レベル                               | 説明                                                                        |
|--------------------------------------|------------------------------------------------------------------------------------|
| Foundation                          | 入門レベル。主要な概念と用語をカバー。                                 |
| Managing Professional (MP)           | IT実務者向け。チーム管理のための実践的/技術的知識に焦点。  |
| Strategic Leader (SL)                | ITとビジネス戦略を整合させるリーダー向け。                                     |
| Practice Manager (PM)                | 特定の管理プラクティスに関するモジュール式の専門トレーニング。                    |
| Extension Modules                    | 持続可能性、クラウド、ビジネス関係、資産管理に関する専門モジュール。 |
| Master                               | 最高レベル。ITILプラクティスの実世界でのリーダーシップと習熟を実証。   |

**認定パス:**

1. **Foundation:**新規参入者とすべてのITスタッフ向け。
2. **Managing Professional / Strategic Leader / Practice Manager:**専門職またはリーダーシップの役割向け。
3. **Master:**組織のITIL卓越性をリードし、実証する人向け。

詳細については、[PeopleCert ITIL 4 Certification](https://www.peoplecert.org/Frameworks-Professionals/ITIL-framework)および[IT Process Wiki on Certification](https://wiki.en.it-processmaps.com/index.php/ITIL_4#ITIL_4_certification_scheme)を参照してください。

## ITILの実装:ステップとベストプラクティス

### 典型的な実装ステップ

1. **評価:**現在のITSMプロセスを分析し、ギャップと改善領域を特定する。
2. **計画:**目標、範囲を定義し、ITIL採用のためのロードマップを作成する。
3. **トレーニング:**ITILトレーニングと認定でチームのスキルを向上させる。
4. **実装:**パイロット領域または影響の大きいプラクティスから始めて、ITILプロセスを展開する。
5. **監視と継続的改善:**KPIを追跡し、結果をレビューし、持続的な改善のために反復する。

([ITIL Best Practices - InvGate](https://invgate.com/itsm/itil/itil-best-practices))

### ベストプラクティス

- **現状から始める:**プロセスを再設計する前に、既存のサービスをマッピングし、ギャップを特定する。
- **サービス指向の考え方を採用する:**顧客とビジネスのための価値創造を優先する。
- **プロセスを標準化する:**反復可能で一貫したワークフローを作成する。
- **SLAを確立する:**サービス提供とパフォーマンスに対する明確な期待を設定する。
- **継続的に改善する:**メトリクスとフィードバックループを使用してプロセスを洗練し、最適化する。
- **サービス卓越性の文化を育成する:**協力、透明性、顧客重視を奨励する。
- **可能な限り自動化する:**繰り返しタスクにワークフロー自動化を使用する。

### 一般的な課題

[Simpliaxis](https://www.simpliaxis.com/resources/itil-implementation-challenges)によると:

1. **経営陣のサポート不足:**ITIL実装にはリーダーシップからの強力な支援とリソースが必要。
2. **ビジョンの欠如:**ITILプラクティスを組織の優先事項と整合させないと、採用が不十分になる。
3. **変更への抵抗:**スタッフは新しいプロセスの採用に消極的な場合がある。コミュニケーションとトレーニングが不可欠。
4. **ITILをプロジェクトとして扱う:**ITILは継続的改善の旅であり、有限のプロジェクトではない。
5. **完璧な瞬間を待つ:**理想的な条件のために実装を遅らせることは、しばしば進歩を停滞させる。

組織は、ステークホルダーを準備し、利点を伝え、適切なトレーニングを提供し、ITILを継続的で進化する取り組みとして扱うことで、これらに対処すべきです。

## ITILの利点と使用事例

### ビジネス上の利点

- **ビジネスとITの整合:**ITサービスがビジネス戦略と成果を推進することを保証する。
- **コスト最適化:**非効率性を特定し、リソース使用を制御し、IT支出を削減する。
- **サービス提供の強化:**プロセスを合理化し、ダウンタイムを削減し、インシデント対応を改善する。
- **顧客満足度の向上:**信頼性が高く、応答性の高いサポートを提供し、SLAを達成する。
- **リスク軽減:**リスクを積極的に管理し、コンプライアンスを確保し、事業継続性をサポートする。

([Ivanti ITIL Guide](https://www.ivanti.com/glossary/itil)、[IBM ITIL Overview](https://www.ibm.com/think/topics/it-infrastructure-library))

### 注目すべき使用事例

| 組織           | 成果                                                                 |
|------------------------|-------------------------------------------------------------------------|
| Procter & Gamble       | 4年間で約5億ドルを節約。ヘルプデスクコールを削減。              |
| Nationwide Insurance   | 停止を40%削減。3年間で430万ドルのROI。             |
| Capital One            | 2年間でビジネスクリティカルなインシデントを92%削減。               |
| 金融サービス企業| 6か月以内にコストを35%以上削減し、生産性を2倍に。          |

**実用的なシナリオ:**

- **インシデントと問題管理:**より迅速な解決、繰り返しインシデントの削減。
- **自動化:**AI駆動のチケット発行、分類、一般的な問題の自己修復。
- **CMDB:**コンプライアンスと効率性のための資産と構成の一元的な追跡。
- **継続的改善:**継続的なプロセスレビューと強化。
- **顧客イニシアチブ:**SLA監視、フィードバックシステム、定期的なレポート。

## 課題と考慮事項

ITILの実装は重要な組織的取り組みです。主な問題には以下が含まれます。

- **トレーニングと認定:**スタッフの教育と認定への継続的な投資。
- **実装タイムライン:**完全な採用には数年かかる場合があり、持続的な焦点が必要。
- **リソース配分:**専任の人員、予算、場合によっては新しいITSMツールが必要。
- **変更管理:**成功のためには文化的および運用的な抵抗を克服することが不可欠。
- **カスタマイズ:**ITILは組織の規模、成熟度、特定のニーズに合わせて調整する必要がある。

([Simpliaxis - ITIL Challenges](https://www.simpliaxis.com/resources/itil-implementation-challenges)、[IT Process Wiki](https://wiki.en.it-processmaps.com/index.php/ITIL_4))

## 現代の方法論との統合

ITIL 4は、Agile、DevOps、AI駆動の自動化などの現代的なアプローチとの統合のために設計されています。

- **Agile:**ITILの反復的改善は、Agileのスプリントベースのサイクルとフィードバックループと整合する。
- **DevOps:**変更、リリース、展開管理のためのITILプラクティスは、DevOpsの継続的デリバリーと協力への焦点を補完する。
- **AIと自動化:**チケットトリアージ、監視、レポートなどの日常的なタスクがますます自動化され、スタッフをより価値の高い作業に解放し、応答時間を改善する。
- **フィードバックと学習:**ITILと現代の方法論の両方が、迅速なフィードバック、学習、継続的改善を強調する。

([ITSM.tools ITIL 4 Explained](https://itsm.tools/itil-4-explained/)、[ITIL Process Wiki](https://wiki.en.it-processmaps.com/index.php/ITIL_4))

|--------------------------------------|-----------------------------------------------------------------------------------------------------------------------------------|

| **ITSM(IT Service Management)**     | ITサービスの設計、提供、管理、改善のための規律。                                                        |
| **継続的サービス改善(CSI)** | より大きな価値のためにITサービスとプロセスを定期的に強化するプロセス。                                                       |

| **問題管理**               | インシデントの再発を防ぐために根本原因を特定し、対処する。                                                        |

| **サービスバリューシステム(SVS)**       | 統合された活動とリソースを通じた価値共創のためのITIL 4の全体的なモデル。                                        |
| **サービスレベル契約(SLA)**    | プロバイダーと顧客間のサービス期待を指定する正式な合意。                                                   |
| **自動化**                       | 最小限の人間の介入でタスクを実行するための技術の使用。                                                               |
| **最適化と自動化**            | 効率性と自動化を提唱するITIL原則。                                                                              |
| **全体的に働く**                | サービスバリューシステムのすべての相互接続を考慮する原則。                                                        |
| **協力と可視性の促進** | チームワーク、透明性、共有知識を強調する原則。                                                             |
| **フィードバックを伴う反復的な進歩** | 段階的な改善を行い、フィードバックを使用して次のステップを導く。                                                             |
| **顧客満足度**            | ITサービスを通じて顧客の期待を一貫して満たすか超える成果。                                           |
